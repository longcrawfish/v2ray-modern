# Phase 1A 审计结论

## 审计范围

- `Dockerfile`
- `caddy.sh`
- `Caddyfile`
- `v2ray.json`
- `v2ray.js`
- `README.md`

---

## 一、目录结构问题

### 1. 顶层文件扁平堆叠，运行时文件、构建文件、配置文件混放

问题：
- 当前仓库顶层直接放置 `Dockerfile`、`caddy.sh`、`Caddyfile`、`v2ray.json`、`v2ray.js`、`index.html`。
- 未区分 `scripts/`、`templates/`、`runtime/`、`exports/`、`infra/` 等职责目录。
- `README.md` 描述的 `compose.yaml`、`scripts/`、`templates/`、`data/` 结构当前并不存在。

风险：
- 后续引入 `ws-tls` 与 `reality` 两种 transport 时，文件命名和职责边界会继续膨胀。
- 公共逻辑与协议逻辑无法拆分，`refactor-base` 分支很容易继续背负协议细节。
- README 与实际结构不一致，使用者和维护者会被误导。

重构建议：
- 在 `refactor-base` 建立最小骨架：`scripts/`、`templates/`、`data/runtime/`、`data/exports/`、`doc/`、`compose.yaml`。
- 将“模板源文件”和“渲染结果”分离，禁止继续把可执行脚本直接写配置正文。
- 顶层只保留少量入口文件，其余下沉到按职责组织的目录。

### 2. 配置生成位置与运行位置耦合

问题：
- `caddy.sh` 直接写 `/etc/Caddyfile`、`/etc/v2ray/config.json`、`/srv/sebs.js`。
- 配置生成、配置落盘、进程启动全部在一个脚本内完成。

风险：
- 后续无法独立做“渲染配置”“校验配置”“仅导出客户端配置”等原子操作。
- 无法支持 compose 场景中的挂载目录、容器外调试、配置复用。

重构建议：
- 公共底座拆出三个阶段：参数装载、模板渲染、进程启动。
- 统一将渲染结果落到 `data/runtime/`，由启动脚本负责复制或挂载到容器约定路径。

## 二、参数系统问题

### 1. 参数来源混乱，未形成统一参数模型

问题：
- `caddy.sh` 使用位置参数 `$1 $2 $3`，分别承载 `domain`、`psname`、`uuid`。
- `README.md` 又声明未来使用 `.env` 中的 `DOMAIN`、`UUID`、`PROFILE`。
- `Dockerfile` 同时混用 `ARG` 与 `ENV`，但这些参数与运行脚本没有形成闭环。

风险：
- 同一个参数在不同层有不同入口，容易出现“镜像构建参数”和“运行参数”语义不一致。
- 无法做参数校验、默认值管理、帮助输出、错误提示。
- 后续新增 transport 专属参数时，位置参数方式会迅速失控。

重构建议：
- 在 `refactor-base` 定义统一参数模型，优先采用 `.env + shell loader` 或统一 CLI 参数解析。
- 公共参数最少应包含：域名、UUID、节点名称、运行模式、输出目录、日志目录。
- transport 专属参数单独命名并分层加载，避免公共底座感知 Reality 或 WS 细节。

### 2. 缺少合法性校验和失败早停

问题：
- `caddy.sh` 对域名、UUID、端口、路径均无校验。
- `README.md` 提到“UUID 必须合法”“80/443 必须未被占用”，但仓库中没有对应检查实现。

风险：
- 运行失败会拖到容器启动阶段才暴露，定位成本高。
- 未来三分支会重复各自实现校验逻辑，造成分叉。

重构建议：
- 公共底座提供 `preflight-check` 能力，只负责校验公共条件与基础依赖。
- transport 分支只补充自身需要的额外校验，例如 WS 路径、Reality 密钥材料等。

## 三、硬编码项清单

### 1. 协议与传输层硬编码

问题：
- `caddy.sh:31` 写死 `protocol: vmess`
- `caddy.sh:41` 写死 `network: ws`
- `caddy.sh:43` 写死 `path: /one`
- `caddy.sh:66` 写死客户端导出 `net: ws`
- `caddy.sh:70` 写死 `tls: tls`

风险：
- 与 `refactor-base` 禁止写协议逻辑直接冲突。
- 任何切到 VLESS / Xray / Reality 的工作都会牵动底座脚本。

重构建议：
- 协议类型、传输类型、路径、TLS 相关字段全部模板化。
- `refactor-base` 仅保留模板注入框架，不保留具体协议默认值。

### 2. 端口、路由、标识硬编码

问题：
- `caddy.sh:16`、`caddy.sh:30` 写死后端端口 `2333`
- `Dockerfile:98` 写死暴露 `443 80 2015`
- `Caddyfile:4` 示例中又写死 `:6555`
- `caddy.sh:69` 写死节点名 `sebsclub`
- `caddy.sh:5` 写死默认 UUID 初值

风险：
- 同一项目内存在多套不一致默认值，容易出现文档、模板、运行行为三方偏差。
- 后续多 profile 并存时无法复用同一底座。

重构建议：
- 公共参数只定义“逻辑端口名”和默认值来源，不在脚本正文散落常量。
- 节点别名、监听端口、导出端口、路径统一进入参数层。

### 3. 实现与供应链硬编码

问题：
- `Dockerfile:28` 写死 `V2RAY_VERSION v4.28.2`
- `Dockerfile:31` 写死下载地址为 V2Ray 官方压缩包
- `Dockerfile:4` 使用 `abiosoft/caddy:builder`
- `Dockerfile:7` 运行时从外部 URL `ADD` builder 脚本
- `Dockerfile:22` 写死维护者信息
- `Dockerfile:25` 写死时区 `Asia/Shanghai`
- `Dockerfile:69` 写死 `ACME_AGREE=false`

风险：
- 底座对 V2Ray 和 Caddy 实现形成强绑定，不利于切换到 Xray 或无 Caddy 模式。
- 构建阶段依赖外部脚本，复现性和可审计性差。

重构建议：
- `refactor-base` 只抽象“核心引擎镜像构建接口”和“反向代理能力接口”，不要把 V2Ray/Caddy 细节写死在底座设计中。
- 供应链相关版本与来源集中管理，后续按分支替换实现。

### 4. 客户端导出格式硬编码

问题：
- `v2ray.js` 与 `caddy.sh` 当前只导出 VMess 订阅信息。
- 文件名 `sebs.js`、文案、二维码输出逻辑均与当前单协议实现耦合。

风险：
- 一旦迁移到 VLESS，公共底座中的导出逻辑将整体失效。
- 未来多个 transport 的客户端导出格式会互相污染。

重构建议：
- 公共底座只提供“导出流程”和目标目录。
- 各 transport 分支提供自己的导出模板和编码器。

## 四、与三分支规划冲突的点

### 1. README 已切到目标态，但仓库实现仍停留在旧架构

问题：
- `README.md` 声明已有 `refactor-base / v2-ws-tls / v2-reality` 工作流。
- 当前实际分支仍为 `master`，且仓库不存在 README 中列出的核心文件。

风险：
- 文档先行但没有落地边界，容易导致开发过程中“按 README 想象编码”。
- 分支职责会在没有底座的情况下提前混入实现细节。

重构建议：
- 先在 `refactor-base` 完成骨架与职责边界，再让 transport 分支在其上增量实现。
- README 需要区分“当前状态”和“目标状态”，避免把规划描述成现状。

### 2. `refactor-base` 禁止协议逻辑，但现入口脚本全部是协议脚本

问题：
- `caddy.sh` 直接生成 `vmess + ws + tls` 相关配置。
- `v2ray.json` 也是单协议样板，不具备公共底座属性。

风险：
- 如果继续以 `caddy.sh` 为基础演进，`refactor-base` 会天然继承协议耦合。

重构建议：
- `refactor-base` 不保留当前脚本作为长期入口，只保留它的行为审计结果。
- 后续公共入口脚本只负责调度，不直接写任何协议字段。

### 3. `v2-reality` 目标要求不依赖 Caddy，但当前架构强依赖 Caddy

问题：
- `Dockerfile` 构建链路围绕 Caddy builder。
- `caddy.sh` 先启动 Caddy，再启动 V2Ray。
- `Caddyfile` 被视为默认存在的配置中心。

风险：
- Reality 分支若沿用现结构，会被迫携带无关组件与端口。
- 镜像、脚本、健康检查都会出现冗余复杂度。

重构建议：
- 将“是否需要反代层”设计为 transport 可选能力。
- 公共底座只约定进程编排接口，不默认反向代理一定存在。

## 五、应下沉到公共底座的逻辑

### 1. 参数装载与默认值管理

问题：
- 参数目前散落在脚本、Dockerfile、README 三处。

风险：
- 同步成本高，容易出现默认值漂移。

重构建议：
- 公共底座统一实现参数读取、默认值、帮助提示、必填校验。

### 2. 配置渲染流程

问题：
- 当前通过 heredoc + `sed` 即席写配置。

风险：
- 不可复用，不可测试，不可组合。

重构建议：
- 公共底座提供模板渲染入口、渲染目录约定、变量注入机制。

### 3. 启动流程编排

问题：
- 当前脚本把配置生成、日志输出、二维码导出、进程拉起全部串在一起。

风险：
- 无法按阶段执行，也不利于 compose 化。

重构建议：
- 公共底座拆分为 `preflight-check`、`render-config`、`start`、`status` 等原子脚本。

### 4. 基础运行约定

问题：
- 日志目录、运行目录、导出目录没有统一契约。

风险：
- 分支之间会各自定义路径，后续运维和排障成本高。

重构建议：
- 在底座约定统一目录和挂载方式，只暴露少量可覆盖参数。

---

## 六、只能留给 transport 分支实现的逻辑

### 1. 协议内核选择

问题：
- V2Ray、Xray、VLESS、VMess 属于协议或核心实现选择。

风险：
- 一旦进入底座，底座就不再中立。

重构建议：
- `v2-ws-tls` 和 `v2-reality` 各自持有核心配置模板与镜像实现。

### 2. 传输层配置

问题：
- WS 路径、TLS 终止、Reality 参数、回落策略都属于 transport 专属逻辑。

风险：
- 公共层一旦感知这些参数，参数体系会迅速变得不稳定。

重构建议：
- transport 分支各自维护 `transport.tpl` 或同类模板片段，由底座负责拼装和渲染。

### 3. 客户端导出格式

问题：
- 不同协议/传输对应不同分享链接、二维码内容、字段映射。

风险：
- 统一导出器会被迫充满条件分支。

重构建议：
- 底座只定义导出命令接口，具体内容由 transport 分支实现。

### 4. 反向代理依赖

问题：
- WS + TLS 可能需要域名与反代层，Reality 不需要 Caddy。

风险：
- 底座内置代理层会让无代理模式变复杂。

重构建议：
- 是否启用 Caddy/Nginx 等代理能力应由 transport 分支声明，而不是底座假定。

---

## 七、结论摘要

- 当前仓库本质上仍是“单脚本 + 单协议 + 单容器入口”的旧式一键部署项目。
- README 已进入 Phase 1A 目标态，但代码尚未完成与目标态一致的底座抽象。
- 最主要的技术债不在某个单点 bug，而在职责混杂：参数、模板、协议、导出、启动被压缩在 `caddy.sh` 与 `Dockerfile` 中。
- `refactor-base` 的首要任务不是替换协议，而是先建立中立底座：参数系统、模板系统、目录契约、compose 化入口、原子脚本。
- `ws-tls` 与 `reality` 的差异应通过 transport 片段承载，而不是继续在公共脚本里写条件分支。
